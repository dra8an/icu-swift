# Islamic Tabular & Civil Regression Testing

## Background

The Islamic Tabular and Islamic Civil calendars are **purely arithmetic** —
fixed Type II 30-year leap cycle, fixed alternating month lengths, no
astronomy. Any correct implementation should agree bit-for-bit with any
other. This makes them an ideal candidate for daily regression against
external sources.

Unlike Hebrew (where Hebcal is the canonical reference) or Chinese (where
the Hong Kong Observatory publishes authoritative tables), there is **no
governmental reference** for the Tabular calendar. No country uses it for
real religious or civil purposes:

- Saudi Arabia uses **Umm al-Qura** (a different astronomical calendar).
- Most Muslim countries use **actual lunar crescent sighting**.
- Iran/Afghanistan use the **Solar Hijri** (Persian) calendar civilly.

The Tabular calendar exists as a deterministic conversion calendar for
scholarly and computer-historical use. Its "authority" is Reingold &
Dershowitz's *Calendrical Calculations* and Kūshyār ibn Labbān's 10th-century
30-year cycle. To validate, we therefore use **two independent code-only
reference implementations** and require they agree with each other and with us.

## Reference Sources

### 1. Foundation `Calendar`

```swift
var iso = Calendar(identifier: .gregorian)
var tbla = Calendar(identifier: .islamicTabular)  // Thursday epoch
var civil = Calendar(identifier: .islamicCivil)   // Friday epoch
```

`Foundation.Calendar` is a thin wrapper over **ICU4C**. Same family as ICU4X
(which we ported from), so this catches *porting* bugs but isn't fully
independent algorithmically.

### 2. Python [`convertdate`](https://pypi.org/project/convertdate/)

A pure-Python independent port of Reingold & Dershowitz's Lisp code.
Different language, different codebase, different maintainers.

```bash
pip3 install --user convertdate
python3 -c "from convertdate import islamic; print(islamic.from_gregorian(2024,4,9))"
# (1445, 9, 30)
```

`convertdate.islamic` implements the **Friday/civil** epoch by default, with
the same Type II leap cycle.

## Epoch Correspondence

| Source                          | Epoch    | Identifier        |
|---------------------------------|----------|-------------------|
| Foundation `.islamicTabular`    | Thursday | `islamic-tbla`    |
| Foundation `.islamicCivil`      | Friday   | `islamic-civil`   |
| `convertdate.islamic`           | Friday   | (unnamed)         |
| Our `IslamicTabular()` (default)| Thursday | `islamic-tbla`    |
| Our `IslamicCivil()`            | Friday   | `islamic-civil`   |

To compare convertdate against the Thursday-epoch (`tbla`) variant, shift
the Gregorian input by **+1 day** before passing it to convertdate. Both
sources then agree with each other and with our implementation across the
full 1900–2100 range.

### Regenerating the CSVs

Both CSVs are generated by a small Swift script using Foundation:

```swift
// gen_islamic_both.swift
import Foundation
var iso = Calendar(identifier: .gregorian); iso.timeZone = TimeZone(identifier: "UTC")!
var tbla = Calendar(identifier: .islamicTabular); tbla.timeZone = TimeZone(identifier: "UTC")!
var civil = Calendar(identifier: .islamicCivil); civil.timeZone = TimeZone(identifier: "UTC")!
let start = iso.date(from: DateComponents(year: 1900, month: 1, day: 1))!
let end   = iso.date(from: DateComponents(year: 2100, month: 12, day: 31))!
var outT = "iso_year,iso_month,iso_day,h_year,h_month,h_day\n"
var outC = outT
var d = start
while d <= end {
    let g  = iso.dateComponents([.year, .month, .day], from: d)
    let ht = tbla.dateComponents([.year, .month, .day], from: d)
    let hc = civil.dateComponents([.year, .month, .day], from: d)
    outT += "\(g.year!),\(g.month!),\(g.day!),\(ht.year!),\(ht.month!),\(ht.day!)\n"
    outC += "\(g.year!),\(g.month!),\(g.day!),\(hc.year!),\(hc.month!),\(hc.day!)\n"
    d = iso.date(byAdding: .day, value: 1, to: d)!
}
let base = "Tests/CalendarAstronomicalTests"
try outT.write(toFile: "\(base)/islamic_tbla_1900_2100.csv", atomically: true, encoding: .utf8)
try outC.write(toFile: "\(base)/islamic_civil_1900_2100.csv", atomically: true, encoding: .utf8)
```

```bash
swift gen_islamic_both.swift
```

### Cross-validating with `convertdate`

```bash
# civil — straight comparison
python3 -c "
from convertdate import islamic
from datetime import date, timedelta
d = date(1900,1,1); end = date(2100,12,31); one = timedelta(days=1)
out = ['iso_year,iso_month,iso_day,h_year,h_month,h_day']
while d <= end:
    y,m,da = islamic.from_gregorian(d.year, d.month, d.day)
    out.append(f'{d.year},{d.month},{d.day},{y},{m},{da}')
    d += one
open('/tmp/cd_civil.csv','w').write('\n'.join(out)+'\n')
"
diff -q /tmp/cd_civil.csv Tests/CalendarAstronomicalTests/islamic_civil_1900_2100.csv

# tbla — shift convertdate input by +1 day
python3 -c "
from convertdate import islamic
from datetime import date, timedelta
d = date(1900,1,1); end = date(2100,12,31); one = timedelta(days=1)
out = ['iso_year,iso_month,iso_day,h_year,h_month,h_day']
while d <= end:
    s = d + one
    y,m,da = islamic.from_gregorian(s.year, s.month, s.day)
    out.append(f'{d.year},{d.month},{d.day},{y},{m},{da}')
    d += one
open('/tmp/cd_tbla.csv','w').write('\n'.join(out)+'\n')
"
diff -q /tmp/cd_tbla.csv Tests/CalendarAstronomicalTests/islamic_tbla_1900_2100.csv
```

Both `diff` invocations report identical files — Foundation and convertdate
agree exactly across all 73,414 days.

## The Regression Test

**File:** `Tests/CalendarAstronomicalTests/IslamicTabularRegressionTests.swift`
**Suite:** `Islamic Tabular / Civil Regression`

A single generic helper handles both calendars:

```swift
private func runRegression<C: CalendarProtocol>(
    csvPath: String,
    calendar: C,
    label: String
) throws -> (checked: Int, failures: Int) where C.DateInner == IslamicTabularDateInner
```

For each row the helper:

1. Parses ISO and Hijri triples.
2. Converts ISO → `RataDie` via `GregorianArithmetic`.
3. Builds `Date<C>.fromRataDie(rd, calendar: calendar)`.
4. Asserts `extendedYear`, `month.ordinal`, and `dayOfMonth` all match.

Two `@Test` methods invoke it once for `IslamicTabular()` (Thursday default)
and once for `IslamicCivil()`.

The tests are gated by file existence — if a CSV is missing the test prints
`SKIP` and passes, so check-outs without the fixtures still build.

## Bugs Caught By This Test

The regression suite caught two real bugs that the prior 33 hand-picked
reference cases had missed for months — see `Docs/Islamic.md` §History for
details:

1. **Calendar mislabeling** — `islamic-tbla` was using the Friday epoch.
2. **Approximate `yearFromFixed`** — off-by-one at end-of-year boundaries
   (104 failures over 200 years), invisible to the 33 reference cases
   because none of them landed on a year boundary.

Both are now fixed; the 0-failure result above is the post-fix state.
